Developing a self transforming brainf*ck compiler

Note: All development is done under and for Linux, None of this applies directly to any other operating system.

Motive:

The research done below is documentation of the process of writing my project: Mindf*ck. A self-transforming brainf*ck compiler (Binary Generator?)

The project requires me to write the entire compiler (Binary Generator?) as a flat binary using x64 assembly (for the sake of efficiency and size). To understand how I would implement my idea for the compiler, I explore how to write binaries that replace their own code with new code. In the end, we figure out how to write a tiny binary so small that we write the code into it's unused ELF header spaces to save binary size.

Writing self transforming binaries in C:

(There are several very good blog posts written by software engineers covering topics on self mutating C code, The sources I used are to be found at the sources section)

(I'm assuming that the reader has a basic understanding of how a program's memory looks like after being loaded into memory by the loader) The text segment of processes in modern x64 computers are protected from being written to, Hence one can only read or execute the code from the segment, In-order to modify code during runtime, we would have to modify the code in the text segment, C lets us change permission for the memory pages (Pages - A page, memory page, or virtual page is a fixed-length contiguous block of virtual memory, described by a single entry in a page table. It is the smallest unit of data for memory management in an operating system that uses virtual memory. Wikipedia) We do this via mprotect(), mprotect requires us to give it a pointer that must be aligned to a page boundary (Page Aligned: The starting address of a block of data or a program segment is a multiple of the system's memory page size).

Example code that moves pointer to the page boundary and then changes the page to read, write, and execute permissions: Source

int change_page_permissions_of_address(void *addr) {
    int page_size = getpagesize();
    addr -= (unsigned long)addr % page_size;

    if(mprotect(addr, page_size, PROT_READ | PROT_WRITE | PROT_EXEC) == -1) {
        return -1;
    }

    return 0;
}

Now this lets us write to the text segment of the process's memory. A simple memcpy() call that writes code over an existing function would create a self mutating C program.

The below code is an example of one such programs which executes a shell. (from source 1)

#include <stdio.h>
#include <unistd.h>
#include <errno.h>
#include <string.h>
#include <sys/mman.h>

void foo(void);
int change_page_permissions_of_address(void *addr);

int main(void) {
    void *foo_addr = (void*)foo;

    // Change the permissions of the page that contains foo() to read, write, and execute
    // This assumes that foo() is fully contained by a single page
    if(change_page_permissions_of_address(foo_addr) == -1) {
        fprintf(stderr, "Error while changing page permissions of foo(): %s\n", strerror(errno));
        return 1;
    }

    puts("Calling foo");
    foo();

    // http://www.exploit-db.com/exploits/13691/
    char shellcode[] =
        "\x48\x31\xd2"                              // xor    %rdx, %rdx
        "\x48\x31\xc0"                              // xor    %rax, %rax
        "\x48\xbb\x2f\x62\x69\x6e\x2f\x73\x68\x00"  // mov    $0x68732f6e69622f2f, %rbx
        "\x53"                                      // push   %rbx
        "\x48\x89\xe7"                              // mov    %rsp, %rdi
        "\x50"                                      // push   %rax
        "\x57"                                      // push   %rdi
        "\x48\x89\xe6"                              // mov    %rsp, %rsi
        "\xb0\x3b"                                  // mov    $0x3b, %al
        "\x0f\x05";                                 // syscall

    // Careful with the length of the shellcode here depending on what is after foo
    memcpy(foo_addr, shellcode, sizeof(shellcode)-1);

    puts("Calling foo");
    foo();

    return 0;
}

void foo(void) {
    int i=0;
    i++;
    printf("i: %d\n", i);
}

Now we do something similar with our brainf*ck compiler, but the problem with writing this in C is, the size of the function foo in the above case for example, is predefined to be 41 bytes, there is no standard way for us to define the size of a function in the text segment with high level languages like C and C++, this restricts the amount of space we can use for our brainf*ck program. C also makes it difficult for us to copy the entire binary into stdout which we can later pipe into a binary.

How the Linux kernel deals with virtual memory addressing:

When multiple processes are run in an OS, the kernel has to map the memory for these processes onto RAM, a hardware component called the Memory Management Unit (MMU) handles this mapping. (To read: Internal workings of MMU, TLBs - Translation Lookaside Buffers)

CPU -- Logical Address --> MMU -- Physical Address --> Physical Memory

The virtual addresses are mapped onto the physical memory as non - contiguous blocks of memory in pages.

How the Linux kernel deals with virtual memory:

The Linux kernel has its own data structure to deal with virtual memory. When the operating system is booted, the bootloader boots the entire kernel into memory.

Modern operating systems usually load the kernel along with initramfs (initial RAM file system) - its a temporary, in-memory file system that the kernel loads during boot to prepare the system before the main root file system is mounted. It also loads some essential kernel modules and drivers. as one user pointed out in a StackOverflow thread: The kernel in most modern Linux setups is heavily module based, i.e., the kernelproper (loaded on boot into RAM) includes just the bare minimum functionality, all the rest is compiled as modules (loadable at runtime).
